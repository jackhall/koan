# Type memos

Content-addressed type identity, memoized dispatch, and modules as values — one
overhaul of `KType`'s identity and satisfaction story. Type identity becomes a wide
content-hash digest replacing raw-pointer nominal identity; subtype-match outcomes are
cached on that digest so overload resolution stops re-walking the same
`(value-type, slot-type)` pair; modules move to the value channel, typed by a
creation-time principal signature under a canonical signature-subtyping relation. The
end state is pinned in
[design/typing/module-values-and-type-identity.md](../../design/typing/module-values-and-type-identity.md).

## Unplanned work

Known gaps adjacent to this project's items, recorded so they aren't lost. **To start
one, first promote it into its own `roadmap/type_memos/<item>.md` file** with
`**Problem.**` / `**Acceptance criteria.**` / `## Dependencies` sections (see the
documentation skill's "When adding a new roadmap item"), then replace its prose entry
here with a link. These paragraphs name the gap, not the fix.

- **Cross-thread type-content transfer.** The type registry is per-run-frame, so when
  future concurrency moves a value between threads, its types' content must land in
  the receiving frame's registry
  ([design/typing/type-registry.md](../../design/typing/type-registry.md),
  "Concurrency"). Two candidate mechanisms are recorded there — copy the value's type
  nodes plus everything reachable through their composition edges, skipping digests
  the receiver already holds; or, with persistent (immutable) node storage, merge the
  maps outright — and whether verdict edges transfer as warm cache is part of the same
  question. Handles need no translation under either: a digest is the same value in
  both registries. Exercisable only once concurrency ships.
- **A registry minted outside the run frame.** `PartialEq for DeferredReturn`
  ([signature.rs](../../src/machine/model/types/signature.rs)) compares expression
  types, which needs a `&TypeRegistry`, but a trait impl takes no parameter — so it
  constructs a cold one per comparison. Answers stay exact (a verdict is a pure cache
  over content digests, and every miss falls through to the structural walk that is the
  source of truth), but it is the one registry in the tree no run frame owns, against
  the home-and-reach rule in
  [design/typing/type-registry.md](../../design/typing/type-registry.md). Whether the
  call site survives [interned type content](interned-type-content.md) — which reshapes
  how type content is reached — decides whether this needs its own fix.
- **Verdict-sharing tests assert the weaker counter.** The cross-module cases in
  `src/builtins/ascribe/tests/self_sig.rs` assert that the registry's hit count rises
  across a second `TAKE`. Measurement shows the miss count is what discriminates: a
  second module with an identical interface leaves misses flat, while one with a
  differing interface turns a hit into a miss — yet the hit delta stays positive in
  both, so the assertion passes even when cross-module sharing is broken. The
  discriminating assertion is on the miss delta.
